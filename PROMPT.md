# Agent 3: Build structural components and migrate the protection page to Web Components

Branch: `refactor/protection-page` (create from main)

## Context

This repo is a design prototype for a car rental booking funnel. We are migrating from BEM-namespaced CSS to vanilla Web Components (Custom Elements + Shadow DOM, no libraries). The `ox-` prefix is used for all custom element tag names; class names use OX PascalCase (e.g. `OXBookingHeader`).

Previous work has established:
- `prototype/components/shared/base-styles.js` — shared constructable stylesheet with typography utilities and icon styling
- `prototype/components/index.js` — central registry that imports all components
- 10 leaf components already built and working: `ox-separator`, `ox-badge`, `ox-price`, `ox-button`, `ox-icon-button`, `ox-text-button`, `ox-chip`, `ox-link`, `ox-list-item`, `ox-floating-button`
- `prototype/styles/all.css` — CSS bundle with all legacy component + page styles

Design tokens from `prototype/styles/tokens.css` are CSS custom properties on `:root` that automatically inherit into Shadow DOM — do NOT embed or duplicate token definitions inside component styles.

Follow the same component authoring pattern used by existing components in `prototype/components/`. Read any existing `ox-*.js` file (e.g. `ox-button.js`) if you need a reference for structure, imports, and conventions.

## Part 1: Build structural components (6 components)

For each component, read its existing CSS file in `prototype/styles/components/` AND its HTML usage in `prototype/pages/protection.html` (and `prototype/pages/offer-list.html` where relevant) to understand the full API and markup structure.

### 1. `ox-radio-button.js` — source: `radio-button.css` + `radio-button.presets.css`
- Class: `OXRadioButton`
- This wraps a native `<input type="radio">` for accessibility and consistent visual styling
- Attributes: `name` (radio group name), `value`, `checked` (boolean), `disabled` (boolean), `error` (boolean)
- The component must contain a visually-hidden native `<input type="radio">` and a styled `.radio-button__control` div
- **Critical for form behavior:** the native radio input must be inside the shadow DOM but radio group behavior (mutual exclusivity by `name`) must work across multiple `<ox-radio-button>` elements. Use one of these approaches:
  - Wire the `name` and `value` attributes onto the native input, and use `ElementInternals` for form association
  - Or: emit a `change` CustomEvent with `{ bubbles: true, composed: true }` when selection changes, so page-level scripts can coordinate radio groups
- Reflect `checked` attribute: when the native input changes, update the host's `checked` attribute
- Support all states from the CSS: default, hover, checked (7px border), focus-visible, disabled, error
- Display: inline-flex

- Slots: default slot for description/body content, or determine from the CSS structure
- Must support all visual variants defined in the CSS

### 3. `ox-protection-card.js` — source: `protection-card.css`
- Class: `OXProtectionCard`
- This is the most complex component in this batch. Read `protection-card.css` (357 lines) AND how cards are generated by the inline `<script>` in `protection.html` very carefully.
- The inline JS in protection.html contains a `renderCards()` function that builds these cards dynamically — study its HTML output to understand the full markup contract.
- Attributes: `option-id`, `option-title`, `selected` (boolean), `expanded` (boolean), `stars` (number 0-3, filled star count), `title`, `deductible`, `deductible-kind` (primary|success|error), `daily-price`, `total-price`, `badge-text` (optional, for discount/recommended badge), `badge-kind` (optional)
- Internal structure (based on existing CSS): top section (secondary-container bg with badge, rating, title, radio, deductible, price) + bottom section (coverage list items, footer text)
- Slots: a `coverage` named slot for the list of coverage items, or render them from a data attribute — study how the existing JS builds them
- Mobile behavior (<=899px): accordion pattern — bottom section collapses/expands with max-height + opacity transition. An accordion toggle button (icon-button with chevron) appears. Staggered fade-in animation on coverage items.
- Desktop behavior (>=900px): always expanded, no accordion, radio shifts to top-right
- Emit custom events: `protection-select` (when card is clicked or radio changes, detail: `{ optionId }`), `protection-toggle` (when accordion expand/collapse is toggled, detail: `{ optionId, expanded }`)
- Display: flex (column)

### 5. `ox-booking-header.js` — source: `booking-header.css`
- Class: `OXBookingHeader`
- This is the MOST COMPLEX component — 284 lines of responsive CSS. Read `booking-header.css` carefully. Also study how it's used differently in `offer-list.html` (with search row) vs `protection.html` (step variant with title row).
- Attributes:
  - `back-href` (URL for back button)
  - `location` (search location text, e.g. "Munich Airport")
  - `dates` (search dates text, e.g. "Mar 16 | 12:00 PM - Mar 20 | 12:00 PM")
  - `step-title` (title text for step pages, e.g. "Which protection package do you need?")
  - `variant` (default|step — "step" maps to `--search-hidden` + `--step` modifiers in the CSS)
  - `theme` (option-light, maps to `data-p100-theme` attribute on the host for token overrides)
- Responsive behavior that must be handled internally:
  - Mobile (<650px): grid layout, back button visible, no logo, icon-only utility links, second row shows search button (default) or step title (step variant). Force light theme colors as the CSS does.
  - Tablet 650-899px: logo + search button + compact utility nav (language as icon, login as text)
  - Desktop >=900px: full layout with all text utility links
- Internal elements: logo SVGs from `../assets/logos/sixt_rent_the_car_on_dark.svg` (reference with path relative to the page, not the component — use a `logo-src` attribute or hardcode the relative path and document it)
- Utility nav links (language, login) can be hardcoded for the prototype
- Uses `layout-shell` for horizontal centering — apply this class on an internal wrapper element, and import the layout CSS rules into the shadow stylesheet (read `prototype/styles/layout.css` and include its rules in the component's shadow styles so the shell works correctly at all breakpoints)

### 6. `ox-booking-footer.js` — source: `booking-footer.css`

- Display: block

## Part 2: Migrate the protection page

After building all 6 components above:

1. **Update `prototype/components/index.js`** — add imports for all 6 new components.

2. **Rewrite `prototype/pages/protection.html`** to use Web Components:
   - Replace the individual CSS `<link>` tags in `<head>` with a single `<link rel="stylesheet" href="../styles/all.css">`
   - Add `<script type="module" src="../components/index.js"></script>`
   - Replace the booking-header BEM markup (~30 lines) with: `<ox-booking-header back-href="./offer-list.html" variant="step" step-title="Which protection package do you need?" theme="option-light"></ox-booking-header>`
   - Replace BEM component markup throughout with Web Component tags: `<ox-button>`, `<ox-text-button>`, `<ox-badge>`, `<ox-separator>`, `<ox-icon-button>`, `<ox-link>`, `<ox-list-item>`, `<ox-price>`, `<ox-banner>`, `<ox-booking-overview>`, `<ox-booking-footer>`
   - Keep page-level layout classes (`layout-shell`, protection-specific classes) and `<link rel="stylesheet" href="../styles/pages/protection.css">` for page composition
   - Keep `<main>`, `<section>`, `<aside>` structural HTML for semantic meaning

3. **Extract the inline `<script>`** (~200 lines) into `prototype/scripts/pages/protection.js`:
   - Move the entire IIFE and all its logic into this external file
   - Reference it from HTML: `<script type="module" src="../scripts/pages/protection.js"></script>`
   - Update DOM queries and event listeners to work with the new Web Component tags:
     - Protection cards may now be `<ox-protection-card>` elements — query by tag name or data attributes
     - Radio selection: listen for `protection-select` custom events from `<ox-protection-card>` elements, or listen for native `change` events that bubble up from `<ox-radio-button>` shadow DOM
     - Price updates: set attributes on `<ox-price>` elements (e.g. `el.setAttribute('integer', '649')`) instead of writing innerHTML with price spans
     - Button enable/disable: set `disabled` attribute on `<ox-button>` elements
     - Banner visibility: toggle `hidden` attribute on `<ox-banner>`
   - The card rendering function (`renderCards`) should create `<ox-protection-card>` elements and set their attributes
   - Keep URL param parsing, state management, and navigation logic intact

4. **Verify the protection page works end-to-end:**
   - Navigation from offer-list.html → protection.html (URL params pass through)
   - Protection cards render with correct data (titles, prices, deductibles, coverage items, stars)
   - Clicking a card selects it (visual 2px border, radio fills)
   - Mobile accordion expand/collapse works
   - Total price updates when selecting different packages
   - Continue button enables after selection
   - Mobile sticky summary bar positions correctly
   - Booking overview sidebar shows correct items
   - Responsive layout works at mobile (<650), tablet (650-899), desktop (>=900)
   - No console errors
   - Navigation to the next step works (continue button)
